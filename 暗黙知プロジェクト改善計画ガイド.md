# 暗黙知プロジェクト改善計画ガイド

タスク・スケジュール管理が機能しないプロジェクトを、継続的に改善するための汎用フレームワーク

---

## 1. 課題構造の分析フレームワーク

### 1-1. 4軸診断

暗黙知プロジェクトの問題は単一の原因ではなく、以下の4軸が複合的に絡み合っている。改善に着手する前に、どの軸が最も機能不全を起こしているかを診断する。

| 軸 | 診断観点 | 機能不全のサイン |
|----|---------|----------------|
| **文化・意識** | 情報共有・管理に対する価値観 | 「管理は余計な仕事」「今まで動いていた」という発言が多い |
| **プロセス** | 開発の進め方が定義・合意されているか | 工程の定義が人によって異なる。手順書が存在しない |
| **ツール** | 情報を記録・共有する仕組みが機能しているか | ツールはあるが誰も使っていない。データが空 |
| **スキル** | 個人がプロセスを実践できる能力があるか | やり方を知らないためツールを使えない。タスク分解ができない |

> **診断の原則：** ツール不足を原因と判断しがちだが、多くの場合「文化・プロセス」の問題がツール形骸化を引き起こしている。ツール導入より先に文化・プロセスの仮説を立てる。

### 1-2. 根本原因の仮説化

4軸診断の結果をもとに「症状→原因→根本原因」の連鎖で仮説を構造化する。

```
【症状】        【原因】              【根本原因（仮説）】

進捗不明    ←  タスクが記録されていない  ←  プロセスが合意されていない
手戻り多発  ←  仕様認識が人によって異なる ←  要件の文書化ルールがない
属人化      ←  情報が個人に閉じている    ←  共有することへの文化的抵抗
改善が定着しない ← 過去の失敗体験       ←  学習性無力感（やっても無駄）
```

## 2. 改善仮説の立て方

### 2-1. 仮説の構造

各根本原因に対して「この対策を打てばこの症状が改善される」という仮説を立てる。仮説は以下の形式で定義する。

```
仮説： [根本原因] が解消されれば、[症状] が [指標] で改善される
対策： [具体的な活動]
検証： [何を見れば仮説が正しいか判断できるか]
```

**仮説の例：**

| # | 根本原因 | 仮説 | 対策 | 検証指標 |
|---|---------|------|------|---------|
| H1 | プロセスが合意されていない | 最小限の工程定義を合意すれば進捗が可視化できる | 工程定義ワークショップ + 1枚のプロセスフロー作成 | チームメンバーが工程を同じ言葉で説明できるか |
| H2 | 要件の文書化ルールがない | テンプレートを用意すれば記録コストが下がり定着する | 要件記述テンプレートの作成と試用 | 1ヶ月後のテンプレート利用率 |
| H3 | 学習性無力感 | 小さな成功体験を積めば改善への抵抗が下がる | 即効性のある自動化（週次レポート等）から始める | メンバーの反応・次の改善提案が出るか |

### 2-2. 優先度の決め方

以下のマトリクスで施策を分類し、「高インパクト × 低コスト」から着手する。

```
          インパクト（小）         インパクト（大）
コスト(低)  ┌─────────────────┬─────────────────┐
           │  やってもいいが    │  最優先で着手    │
           │  後回しでもよい    │  （Quick Win）   │
コスト(高)  ├─────────────────┼─────────────────┤
           │  やらない         │  慎重に計画して  │
           │                  │  中長期で取り組む │
           └─────────────────┴─────────────────┘
```

> **原則：** 最初の1〜2ヶ月は「Quick Win」だけに集中する。チームが「改善で楽になった」と感じることが最大の推進力になる。

## 3. 改善活動の進め方

### 3-1. Phase設計の原則

一度に全てを変えようとしない。以下の4フェーズで段階的に積み上げる。

| Phase | 目的 | 期間の目安 | 成功の定義 |
|-------|------|-----------|-----------|
| **Phase 1: 可視化** | 「今どうなっているか」を把握する | 1〜2週間 | 現状の課題が全員に見える状態 |
| **Phase 2: 最小プロセス定義** | 「どう動くか」を最小限合意する | 2〜4週間 | 1枚のプロセスフローをチームが共有している |
| **Phase 3: 定着化** | 合意したプロセスを継続できる仕組みにする | 1〜2ヶ月 | 推進者がいなくてもプロセスが回る |
| **Phase 4: 改善サイクル** | 自律的に改善し続ける組織にする | 3ヶ月〜 | チームが自分たちで問題を発見・改善できる |

### 3-2. 各Phaseの活動内容

#### Phase 1: 可視化

**目標：** 現状の実態データを収集し、問題を「感覚」から「事実」に変える。

- Gitコミット履歴の分析（誰が何を触っているか）
- 既存ドキュメント（Excel・議事録等）の棚卸し
- メンバーへの個別ヒアリング（責める場ではなく現状把握の場）
- 課題の一覧化と共有（診断結果レポートの作成）

> **注意：** この段階で「犯人探し」にならないよう、問題はプロセスにあるという前提でヒアリングを設計する。

#### Phase 2: 最小プロセス定義

**目標：** チームが合意できる「最小限のルール」を作る。完璧を目指さない。

- プロセスワークショップの実施（1〜2時間、全員参加）
- 工程の定義（要件 → 設計 → 実装 → テスト の各入口・出口条件）
- タスク管理の最小ルール（どこに書く・誰が更新する・いつ確認する）
- 外部委託先への指示テンプレートの作成

> **原則：** ルールは「守れる最小」に留める。理想のプロセスではなく、今のチームが実践できるプロセスを定義する。

#### Phase 3: 定着化

**目標：** 推進者への依存を排除し、プロセスが自律的に回る仕組みを作る。

- 定期レビューの設計（週次/月次のリズムを作る）
- 自動化の導入（週次レポート自動生成・アラート通知等）
- チェックリストの整備（属人的な判断をチェックリスト化）
- 定着状況のモニタリングと軌道修正

#### Phase 4: 改善サイクル

**目標：** チームが自分たちでプロセスを改善し続ける文化を作る。

- 振り返り（レトロスペクティブ）の習慣化
- プロセス改善提案の仕組みの整備
- ナレッジの蓄積と共有（今回整備したドキュメントの更新継続）
- 新メンバーのオンボーディングへの組み込み

### 3-3. 施策の優先順位付け

Phase内での施策優先順位は以下の基準で決める。

1. **即効性** — 1〜2週間以内に効果が見えるか
2. **依存関係** — 他の施策の前提になっているか
3. **抵抗の少なさ** — チームが受け入れやすいか
4. **自動化可能性** — 人間の継続行動に頼らず仕組み化できるか

## 4. 継続運用の設計

### 4-1. スキルレベル別の実践方法

チームメンバーのスキルレベルに応じて、求めるアクションの粒度を変える。一律に高いスキルを前提とした運用設計は定着しない。

| レベル | 特徴 | 求めるアクション | 支援方法 |
|-------|------|----------------|---------|
| **Lv1: 初級** | プロセス・ツールの経験が少ない | チェックリストに沿って動く | 手順書・テンプレートを整備する |
| **Lv2: 中級** | 自分の作業は管理できるが、全体視点が弱い | タスクを自分でIssue化・更新できる | フォーマットを統一し判断コストを下げる |
| **Lv3: 上級** | プロセス全体を理解し他者を支援できる | プロセスの問題を発見・提案できる | 振り返りの場でリーダー役を担う |

> **設計の原則：** Lv1のメンバーが「考えなくても動ける」設計を基準にする。Lv3向けの高度な運用は後から積み上げる。

### 4-2. 定着化のメカニズム

以下の3要素が揃うと、プロセスが自律的に継続する。

```
① やることが明確（曖昧さの排除）
      ↓
② やった結果が見える（フィードバックループ）
      ↓
③ やらないと困る仕組みになっている（構造的強制力）
```

**具体的な施策例：**

| 要素 | 施策例 |
|------|-------|
| やることが明確 | チェックリスト、テンプレート、ルールの文書化 |
| 結果が見える | 週次レポートの自動生成・共有、ダッシュボード |
| 構造的強制力 | レビュー通過条件にIssue登録を含める、定例会議での確認 |

### 4-3. 改善の振り返り方法

月次または四半期で以下を確認し、計画を軌道修正する。

**振り返りの観点：**

- **プロセス遵守率** — ルールが守られているか（Issue登録率・更新頻度等）
- **問題の早期発見** — 遅延・手戻りが早めに検知できるようになったか
- **チームの感触** — メンバーが「楽になった」と感じているか
- **仮説の検証結果** — 立てた仮説は正しかったか、修正が必要か

> **振り返りの注意点：** 数値だけで判断しない。「なぜ守られなかったか」をプロセス・ルールの問題として捉え、個人を責める場にしない。

### 4-4. スキルアップのロードマップ

プロセス改善と並行して、チームのスキルを段階的に引き上げる。

```
Month 1-2:  【基礎定着】
            チェックリスト・テンプレートに沿って動ける
            Issueの登録・更新が習慣になる

Month 3-4:  【自律化】
            自分でタスクを分解してIssue化できる
            進捗を自分から報告する習慣が生まれる

Month 5-6:  【改善参加】
            プロセスの問題に自分で気づき提案できる
            他メンバーをサポートできる

Month 6以降: 【自走】
            推進者なしでプロセスが維持・改善される
```

## 付録：改善推進者へのチェックリスト

### やるべきこと

- [ ] 問題をプロセスの問題として定義し、個人の責任にしない
- [ ] 最初の成果を2週間以内に出し、チームに見せる
- [ ] ルールは「守れる最小」から始め、徐々に積み上げる
- [ ] 自動化できるものは自動化し、人間の継続努力に頼らない
- [ ] 自分（推進者）が実質的なPMにならないよう役割を限定する

### 避けるべきこと

- [ ] 一度に全プロセスを変えようとする
- [ ] 理想のツール・プロセスを完璧な形で導入しようとする
- [ ] 過去の失敗を理由にした抵抗を無視して強行する
- [ ] 推進者だけが情報を把握し、他は何も分からない状態を作る
- [ ] 改善の成果を推進者個人の手柄にする（チームの成果として扱う）
