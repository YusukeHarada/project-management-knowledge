# 暗黙知プロジェクト改善計画ガイド

タスク・スケジュール管理が機能しないプロジェクトを、継続的に改善するための汎用フレームワーク

## このガイドの対象読者

| 読者 | 使い方 |
|------|-------|
| **推進者**（プロセス改善担当者） | 全体を通して読む。分析・仮説・実行計画の設計に使う |
| **PM・PL** | § 4「改善活動設計」と § 5「継続運用設計」を中心に読む |
| **上位報告・承認者（部長等）** | [標準との対応マップ](標準との対応マップ.md) と組み合わせて活用する |

> **「推進者」とは：** プロセス改善を専任または兼任で担当する人。社内コンサルに近い立ち位置。PMの代替者ではなく、仕組みを作りPMとPLをサポートする役割。PMが自分自身でこのガイドを使う場合も「推進者」として読む。

> **「G0（ガバナンス確立）」とは：** このガイド全体を通して最初に行うべきステップ。改善を承認・強制する権限を持つ上位者（部長等）をスポンサーとして確定し、PMへの指示を通じて「やらざるを得ない構造」を作ること。G0なしに改善を進めると推進者がPM化するだけで組織は変わらない。詳細は § 4-1「Phase 0：ガバナンス確立」を参照。

## 1. 課題構造の分析フレームワーク

### 1-1. 4軸診断

暗黙知プロジェクトの問題は単一の原因ではなく、以下の4軸が複合的に絡み合っている。改善に着手する前に、どの軸が最も機能不全を起こしているかを診断する。

| 軸 | 診断観点 | 機能不全のサイン |
|----|---------|----------------|
| **ガバナンス** | 改善を承認・強制する権限構造が機能しているか | PMが機能していない。やらなくても誰も困らない。責任の所在が曖昧 |
| **文化・意識** | 情報共有・管理に対する価値観 | 「管理は余計な仕事」「今まで動いていた」という発言が多い |
| **プロセス** | 開発の進め方が定義・合意されているか | 工程の定義が人によって異なる。手順書が存在しない |
| **ツール** | 情報を記録・共有する仕組みが機能しているか | ツールはあるが誰も使っていない。データが空 |
| **スキル** | 個人がプロセスを実践できる能力があるか | やり方を知らないためツールを使えない。タスク分解ができない |

> **診断の原則：** ツール・プロセスの問題と診断しがちだが、その奥に「ガバナンスの不在」が隠れていることが多い。仕組みを整えても動かない場合は、ガバナンス軸を最初に疑う。

> **ガバナンスと他の軸の違い：** 文化・プロセス・ツール・スキルの問題は「やり方が分からないから動けない」。ガバナンスの問題は「やらなくても困らないから動かない」。解決策が根本的に異なる。

### 1-2. 「プロセス」の定義

本ガイドにおける「プロセス」とは、**作業の進め方・手順・ルールをチーム全員が共通認識として持っている状態**を指す。具体的には以下の3要素を含む。

| 要素 | 内容 | 不在のサイン |
|------|------|------------|
| **工程の定義** | 各工程の開始条件・完了条件が定まっている | 「完了」の意味が人によって異なる |
| **作業ルールの定義** | 誰が・何を・いつ・どこに記録するかが決まっている | タスクが誰かの頭の中にしかない |
| **判断基準の定義** | 進捗・遅延・完了を判断する共通基準がある | 進捗を聞く人によって答えが変わる |

#### 工程定義の5つの問い

各工程を定義する際には、以下の5つの問いに答える必要がある。この5つが合意されていない工程は「暗黙知」の温床になる。

| 問い | 内容 | 別名 |
|------|------|------|
| **何をもって始まるか** | この作業を開始できる条件は何か | 開始条件（Entry Criteria） |
| **何が必要か** | 作業に必要なインプット・情報・リソースは何か | インプット |
| **何を決めないといけないか** | 作業中に判断・合意が必要なことは何か | 判断ポイント |
| **何をもって完了とするか** | 完了を宣言できる条件は何か | 完了条件（Exit Criteria） |
| **その条件は誰がどう決めるか** | 開始・完了の判断権限は誰にあるか | 承認者・判断プロセス |

> **ポイント：** 「なんとなく進んでいる」から「合意に基づいて進んでいる」に変えることが、プロセス定義の本質的な目的である。

### 1-3. 根本原因の仮説化

4軸診断の結果をもとに「症状→原因→根本原因」の連鎖で仮説を構造化する。

```
【症状】        【原因】                    【根本原因（仮説）】

進捗不明    ←  タスクが記録されていない    ←  記録する場所・タイミングが未定義
            ←  完了の定義が人によって異なる ←  工程定義の5つの問いが未合意
手戻り多発  ←  仕様認識が人によって異なる  ←  要件の文書化ルールがない
属人化      ←  情報が個人に閉じている      ←  共有することへの文化的抵抗
改善が定着しない ← やらなくても困らない    ←  ガバナンスの不在（最重要）
            ←  過去の改善失敗体験          ←  学習性無力感（やっても無駄）
```

> **重要：** 「改善が定着しない」の根本原因がガバナンス不在の場合、プロセス・ツール・スキルをいくら整備しても解決しない。先にガバナンスを設計しなければ、改善活動自体が空振りに終わる。

## 2. 改善仮説の立て方

### 2-1. 仮説の構造

各根本原因に対して「この対策を打てばこの症状が改善される」という仮説を立てる。仮説は以下の形式で定義する。

```
仮説： [根本原因] が解消されれば、[症状] が [指標] で改善される
対策： [具体的な活動]
検証： [何を見れば仮説が正しいか判断できるか]
```

**仮説の例：**

> **前提：** 5軸診断でガバナンス不在が確認された場合、G0を最初に立てる。G0が解消されない限り、H1以降の仮説を実行しても定着しない。

| # | 根本原因 | 仮説 | 対策 | 検証指標 |
|---|---------|------|------|---------|
| G0 | ガバナンスの不在 | 上位職がスポンサーとして機能すれば、チームに「やらざるを得ない構造」が生まれる | 上位職への現状損失の可視化・スポンサー合意 | 上位職からPMへの改善指示が出たか |
| H1 | プロセスが合意されていない | 最小限の工程定義を合意すれば進捗が可視化できる | 工程定義ワークショップ + 1枚のプロセスフロー作成 | チームメンバーが工程を同じ言葉で説明できるか |
| H2 | 要件の文書化ルールがない | テンプレートを用意すれば記録コストが下がり定着する | 要件記述テンプレートの作成と試用 | 1ヶ月後のテンプレート利用率 |
| H3 | 学習性無力感 | 小さな成功体験を積めば改善への抵抗が下がる | 即効性のある自動化（週次レポート等）から始める | メンバーの反応・次の改善提案が出るか |

### 2-2. メトリクスの定義

仮説を立てたら、**改善策の実施前に**「何をもって改善できたか」を測るメトリクスを定義する。メトリクスが未定義のまま実施すると、評価が「感覚」で終わり、次の改善につながらない。

**メトリクス定義で決める5項目：**

| 項目 | 内容 | 例 |
|------|------|-----|
| **何を測るか** | 改善対象を数値化できる指標 | Issue更新頻度・手戻り件数・遅延検知までの日数 |
| **どうやって測るか** | 計測方法・データの取得元 | GitLabのログ・週次レポート・レビュー記録 |
| **いつ測るか** | 計測のタイミング・頻度 | 週次・月次・マイルストーン時点 |
| **何をもって改善とするか** | 目標値・閾値 | Issue更新率80%以上・手戻り件数50%削減 |
| **誰が測るか** | 計測の責任者 | 推進者・PL・自動集計 |

> **重要：** 現状調査のフェーズで**ベースライン（改善前の現状値）を必ず記録する**。ベースラインがなければ改善後との比較ができず、評価が成立しない。

**メトリクス定義テンプレート：**

```
仮説No：[H1 / H2 / ...]

測定指標：
計測方法：
計測頻度：
ベースライン（現状値）：       ← 現状調査フェーズで記入
目標値：
計測責任者：
```

### 2-3. 優先度の決め方

以下のマトリクスで施策を分類し、「高インパクト × 低コスト」から着手する。

```
          インパクト（小）         インパクト（大）
コスト(低)  ┌─────────────────┬─────────────────┐
           │  やってもいいが    │  最優先で着手    │
           │  後回しでもよい    │  （Quick Win）   │
コスト(高)  ├─────────────────┼─────────────────┤
           │  やらない         │  慎重に計画して  │
           │                  │  中長期で取り組む │
           └─────────────────┴─────────────────┘
```

> **原則：** 最初の1〜2ヶ月は「Quick Win」だけに集中する。チームが「改善で楽になった」と感じることが最大の推進力になる。

## 3. ステークホルダーの巻き込み方

改善が失敗する原因の多くは、技術・プロセスの問題ではなく**人・組織の合意形成の失敗**である。改善策を設計する前に、誰の合意が必要で、どう働きかけるかを整理しておく。

### 3-1. ステークホルダーの特定

プロセス改善に関わるステークホルダーを以下の4タイプに分類し、それぞれの関与レベルを把握する。

| タイプ | 役割 | 関与レベル | 主な懸念 |
|-------|------|-----------|---------|
| **スポンサー** | 改善活動を承認・支援する経営層・PM | 意思決定・予算確保 | コスト・効果・リスク |
| **推進協力者** | 改善を一緒に進めるPL・リードエンジニア | 実務での実践・展開 | 自分の作業が増えないか |
| **実践者** | 改善されたプロセスを日常的に使うメンバー | プロセスの遵守 | 使いにくくないか・評価に影響しないか |
| **外部関係者** | 外部委託先・他部署など | 情報共有・連携 | 指示が明確になるか |

### 3-2. 合意形成のタイミングと順序

合意は「上から下へ」の順序で進める。現場だけで合意しても、上位承認がなければ改善が頓挫する。

```
① スポンサーの承認
   └ 改善の必要性・方針・期待効果を説明し、推進の承認を得る
   └ タイミング：改善活動の開始前

② 推進協力者（PL等）との合意
   └ 課題認識を共有し、改善の方向性を一緒に決める
   └ タイミング：スポンサー承認後、実施計画の設計段階

③ 実践者への展開
   └ 「なぜやるか」を丁寧に説明し、小さな成功体験を共有する
   └ タイミング：Phase 2（最小プロセス定義）の実施前

④ 外部関係者への周知
   └ 変更点と影響を明確に伝える
   └ タイミング：プロセス変更が外部に影響するタイミングで都度
```

### 3-3. 抵抗への対処

改善への抵抗は「悪意」ではなく「合理的な懸念」から生まれることがほとんどである。抵抗のパターンと対処法を事前に把握しておく。

| 抵抗のパターン | 背景にある懸念 | 対処法 |
|--------------|--------------|-------|
| 「今まで動いていた」 | 現状変更のリスクへの不安 | 小さく試す・失敗しても戻せることを示す |
| 「入力が面倒」 | 作業量が増えることへの抵抗 | テンプレート・自動化で入力コストを下げる |
| 「監視されている」 | 個人評価への影響を警戒 | プロセスの問題であり個人評価と無関係と明言する |
| 「効果が見えない」 | 改善の成果への懐疑 | 2週間以内に小さな成果を見せる（Quick Win） |
| 「自分には関係ない」 | 当事者意識の欠如 | 課題が自分にどう影響しているかを具体的に示す |

> **原則：** 抵抗を「無視して強行する」のではなく、「懸念を解消する設計に変える」ことで乗り越える。

### 3-4. スポンサーが機能していない場合の戦略

PMや直属の上長がスポンサーとして機能しない場合、推進者がチームに直接働きかけても消耗するだけで組織は変わらない。この場合は**上位職への働きかけ**を優先する。

```
❌ 機能しないアプローチ
推進者 → チーム・PM に直接働きかける
  結果：抵抗・無視・丸投げ → 推進者が消耗

⭕ 機能するアプローチ
推進者 → 上位職に「現状の損失」をデータで見せる
  上位職 → PMに「やれ」と指示する
  PM    → PLに展開せざるを得ない
  結果：構造的にやらざるを得ない状態が生まれる
```

**上位職への働きかけ方：**

| ステップ | 内容 |
|---------|------|
| **1. 現状の損失を可視化する** | 進捗不明・手戻り・属人化が組織にどれだけのコスト・リスクをもたらしているかをデータで示す |
| **2. 感情ではなく事実で話す** | 「誰かが悪い」ではなく「プロセスの問題でこういう損失が起きている」と伝える |
| **3. 上位職に求めることを明確にする** | 「PMへの指示」「改善活動の承認」など、具体的なアクションを依頼する |
| **4. 推進者の役割範囲を合意する** | 「推進者は基盤整備をする。実行はチームがする」という境界を明確にしてもらう |

### 3-5. 推進者が陥りやすい罠

- **推進者が実質的なPMになる** — 情報が推進者に集中し、改善基盤が個人資産になる。「ツールを作る人」と「ツールを使う人」の境界を明確にする
- **やらない人を動かそうとする** — ガバナンス不在の組織で推進者が直接説得を試みても消耗するだけ。動かす主体は上位職であり、推進者は「問題の可視化」に徹する
- **合意なしに展開する** — 「良いことだから分かるはずだ」という思い込みで進めると、後から強い抵抗を受ける
- **成果を推進者の手柄にする** — 改善の成果はチームの成果として共有し、組織の文化として定着させる

## 4. 改善活動の進め方

### 4-0. 改善サイクルの全体像

ソフトウェアプロセス改善は一度で完結するものではなく、以下のサイクルを継続的に回し続けることで組織が成熟していく。

```
        ┌─────────────────────────────────┐
        │        0. 現状理解              │
        │  プロジェクト・組織の実態把握    │
        └────────────────┬────────────────┘
                         ↓
        ┌─────────────────────────────────┐
        │        1. 課題定義              │
        │  症状の整理 → 根本原因の仮説化  │
        └────────────────┬────────────────┘
                         ↓
        ┌─────────────────────────────────┐
        │        2. 現状調査              │
        │  仮説を裏付けるデータ・事実収集  │
        │  ベースライン（現状値）の記録    │
        └────────────────┬────────────────┘
                         ↓
        ┌─────────────────────────────────┐
        │        3. 改善策の設計          │
        │  対策の立案・優先付け           │
        │  メトリクスの定義               │
        └────────────────┬────────────────┘
                         ↓
        ┌─────────────────────────────────┐
        │        4. 実施                  │
        │  小さく試す → 段階的に展開      │
        │  計測の開始                     │
        └────────────────┬────────────────┘
                         ↓
        ┌─────────────────────────────────┐
        │        5. 評価                  │
        │  メトリクスでベースラインと比較  │
        │  仮説は正しかったか判断         │
        └────────────────┬────────────────┘
                         ↓
        ┌─────────────────────────────────┐
        │        6. 次の改善へ            │
        │  結論から新たな課題・仮説を立てる│
        └────────────────┬────────────────┘
                         ↓
                   （1に戻る）
```

**サイクルを回す上での2つの原則：**

- **仮説を立ててから調査する** — 先に仮説を立てることで「何を調べるべきか」が明確になる。仮説なしの調査はデータが集まっても結論が出ない
- **小さく試してから展開する** — 対策を全面展開する前に一部のチーム・工程で試験的に実施し、失敗コストを最小化する

### 4-1. Phase設計の原則

一度に全てを変えようとしない。以下のフェーズで段階的に積み上げる。

> **ガバナンス不在の場合は必ずPhase 0から始める。** Phase 0が完了しないままPhase 1以降に進むと、推進者がPM化するだけで組織は変わらない。

| Phase | 目的 | 期間の目安 | 成功の定義 | 適用条件 |
|-------|------|-----------|-----------|---------|
| **Phase 0: ガバナンス確立** | 「やらざるを得ない構造」を作る | 1〜2週間 | 上位スポンサーの承認・PMへの指示が出た状態 | ガバナンス不在の場合のみ（必須） |
| **Phase 1: 可視化** | 「今どうなっているか」を把握する | 1〜2週間 | 現状の課題が全員に見える状態 | 全ケース |
| **Phase 2: 最小プロセス定義** | 「どう動くか」を最小限合意する | 2〜4週間 | 1枚のプロセスフローをチームが共有している | 全ケース |
| **Phase 3: 定着化** | 合意したプロセスを継続できる仕組みにする | 1〜2ヶ月 | 推進者がいなくてもプロセスが回る | 全ケース |
| **Phase 4: 改善サイクル** | 自律的に改善し続ける組織にする | 3ヶ月〜 | チームが自分たちで問題を発見・改善できる | 全ケース |

### 4-2. 各Phaseの活動内容

#### Phase 0: ガバナンス確立（ガバナンス不在の場合のみ）

**目標：** 上位スポンサーの承認を得て「やらざるを得ない構造」を作る。これが完了するまでPhase 1に進まない。

- 現状の損失をデータで可視化した資料の作成（コスト・品質・組織リスク）
- 上位職（スポンサー候補）への報告・承認依頼
- 推進者の役割範囲の公式合意（推進者はPM代替ではなく基盤整備者）
- スポンサーからPMへの改善指示の確認

> **注意：** Phase 0の成功基準は「スポンサーの言葉の承認」ではなく「PMへの具体的な指示が出たこと」。指示なき承認は機能しない。

#### Phase 1: 可視化

**目標：** 現状の実態データを収集し、問題を「感覚」から「事実」に変える。

- Gitコミット履歴の分析（誰が何を触っているか）
- 既存ドキュメント（Excel・議事録等）の棚卸し
- メンバーへの個別ヒアリング（責める場ではなく現状把握の場）
- 課題の一覧化と共有（診断結果レポートの作成）

> **注意：** この段階で「犯人探し」にならないよう、問題はプロセスにあるという前提でヒアリングを設計する。

#### Phase 2: 最小プロセス定義

**目標：** チームが合意できる「最小限のルール」を作る。完璧を目指さない。

- プロセスワークショップの実施（1〜2時間、全員参加）
- 工程ごとに「工程定義の5つの問い」を使って開始条件・完了条件を合意する（下記テンプレート参照）
- タスク管理の最小ルール（どこに書く・誰が更新する・いつ確認する）
- 外部委託先への指示テンプレートの作成

**工程定義テンプレート（各工程で記入する）：**

```
工程名：[例：要件定義 / 基本設計 / 実装 / テスト]

1. 何をもって始まるか（開始条件）
   →

2. 何が必要か（インプット）
   →

3. 何を決めないといけないか（判断ポイント）
   →

4. 何をもって完了とするか（完了条件）
   →

5. その条件は誰がどう決めるか（承認者・判断プロセス）
   →
```

> **原則：** ルールは「守れる最小」に留める。理想のプロセスではなく、今のチームが実践できるプロセスを定義する。

#### Phase 3: 定着化

**目標：** 推進者への依存を排除し、プロセスが自律的に回る仕組みを作る。

- 定期レビューの設計（週次/月次のリズムを作る）
- 自動化の導入（週次レポート自動生成・アラート通知等）
- チェックリストの整備（属人的な判断をチェックリスト化）
- 定着状況のモニタリングと軌道修正

#### Phase 4: 改善サイクル

**目標：** チームが自分たちでプロセスを改善し続ける文化を作る。

- 振り返り（レトロスペクティブ）の習慣化
- プロセス改善提案の仕組みの整備
- ナレッジの蓄積と共有（今回整備したドキュメントの更新継続）
- 新メンバーのオンボーディングへの組み込み

### 4-3. 施策の優先順位付け

Phase内での施策優先順位は以下の基準で決める。

1. **即効性** — 1〜2週間以内に効果が見えるか
2. **依存関係** — 他の施策の前提になっているか
3. **抵抗の少なさ** — チームが受け入れやすいか
4. **自動化可能性** — 人間の継続行動に頼らず仕組み化できるか

## 5. 継続運用の設計

### 5-1. スキルレベル別の実践方法

チームメンバーのスキルレベルに応じて、求めるアクションの粒度を変える。一律に高いスキルを前提とした運用設計は定着しない。

| レベル | 特徴 | 求めるアクション | 支援方法 |
|-------|------|----------------|---------|
| **Lv1: 初級** | プロセス・ツールの経験が少ない | チェックリストに沿って動く | 手順書・テンプレートを整備する |
| **Lv2: 中級** | 自分の作業は管理できるが、全体視点が弱い | タスクを自分でIssue化・更新できる | フォーマットを統一し判断コストを下げる |
| **Lv3: 上級** | プロセス全体を理解し他者を支援できる | プロセスの問題を発見・提案できる | 振り返りの場でリーダー役を担う |

> **設計の原則：** Lv1のメンバーが「考えなくても動ける」設計を基準にする。Lv3向けの高度な運用は後から積み上げる。

### 5-2. 定着化のメカニズム

以下の3要素が揃うと、プロセスが自律的に継続する。

```
① やることが明確（曖昧さの排除）
      ↓
② やった結果が見える（フィードバックループ）
      ↓
③ やらないと困る仕組みになっている（構造的強制力）
```

**具体的な施策例：**

| 要素 | 施策例 |
|------|-------|
| やることが明確 | チェックリスト、テンプレート、ルールの文書化 |
| 結果が見える | 週次レポートの自動生成・共有、ダッシュボード |
| 構造的強制力 | レビュー通過条件にIssue登録を含める、定例会議での確認 |

### 5-3. 改善の振り返り方法

月次または四半期で以下を確認し、計画を軌道修正する。

**振り返りの観点：**

- **プロセス遵守率** — ルールが守られているか（Issue登録率・更新頻度等）
- **問題の早期発見** — 遅延・手戻りが早めに検知できるようになったか
- **チームの感触** — メンバーが「楽になった」と感じているか
- **仮説の検証結果** — 立てた仮説は正しかったか、修正が必要か

> **振り返りの注意点：** 数値だけで判断しない。「なぜ守られなかったか」をプロセス・ルールの問題として捉え、個人を責める場にしない。

### 5-4. スキルアップのロードマップ

プロセス改善と並行して、チームのスキルを段階的に引き上げる。

```
Month 1-2:  【基礎定着】
            チェックリスト・テンプレートに沿って動ける
            Issueの登録・更新が習慣になる

Month 3-4:  【自律化】
            自分でタスクを分解してIssue化できる
            進捗を自分から報告する習慣が生まれる

Month 5-6:  【改善参加】
            プロセスの問題に自分で気づき提案できる
            他メンバーをサポートできる

Month 6以降: 【自走】
            推進者なしでプロセスが維持・改善される
```

## 付録：改善推進者へのチェックリスト

### やるべきこと

- [ ] 問題をプロセスの問題として定義し、個人の責任にしない
- [ ] 最初の成果を2週間以内に出し、チームに見せる
- [ ] ルールは「守れる最小」から始め、徐々に積み上げる
- [ ] 自動化できるものは自動化し、人間の継続努力に頼らない
- [ ] 自分（推進者）が実質的なPMにならないよう役割を限定する

### 避けるべきこと

- [ ] 一度に全プロセスを変えようとする
- [ ] 理想のツール・プロセスを完璧な形で導入しようとする
- [ ] 過去の失敗を理由にした抵抗を無視して強行する
- [ ] 推進者だけが情報を把握し、他は何も分からない状態を作る
- [ ] 改善の成果を推進者個人の手柄にする（チームの成果として扱う）
