# 暗黙知プロジェクト改善計画ガイド（具体適用版）

汎用フレームワーク（改善計画ガイド）を、対象プロジェクトの実態に適用した実践版

---

## 1. プロジェクト現状診断

### 1-1. プロジェクト概要

| 項目 | 内容 |
|------|------|
| チーム構成 | PM × 1、チーム × 4（制御PF・Linux PF・セキュリティ・OTA）、各チームPL × 1・開発者 × 3・外部委託先 |
| 使用ツール | Git・GitLab・Teams・MS Office |
| 開発フェーズ | 明確な工程定義なし（要件〜テストが混在・非定義） |
| 過去の改善 | 進捗管理を試みたが形骸化・現在は何も実施されていない |

### 1-2. 5軸診断の適用結果

| 軸 | 診断結果 | 深刻度 |
|----|---------|-------|
| **ガバナンス** | PMが機能不全。改善への指示・承認なし。やらなくても誰も困らない構造 | ★★★★★ |
| **文化・意識** | 自責意識なし。改善を推進者に丸投げしようとする意識が支配的 | ★★★★★ |
| **プロセス** | 工程定義なし。要件〜テストが口頭・感覚で進んでいる | ★★★★☆ |
| **ツール** | GitLabは存在するがIssue管理は未使用。ExcelとTeamsが中心 | ★★★☆☆ |
| **スキル** | プロセス管理経験が乏しい。タスク分解・進捗管理スキルが不足 | ★★★☆☆ |

> **診断の結論：** ガバナンスと文化の問題が最も深刻。プロセス・ツール・スキルを整備しても、ガバナンスが機能しない限り定着しない。改善の起点はガバナンス設計に置く。

### 1-3. 根本原因の仮説

```
【症状】              【原因】                      【根本原因（仮説）】

進捗不明          ←  タスクが記録されていない      ←  記録ルールが未定義
                  ←  完了定義が人によって異なる     ←  工程定義の合意なし

手戻り多発        ←  仕様認識が人によって異なる    ←  要件の文書化ルールがない
                  ←  チーム間の依存関係が見えない   ←  チーム横断の進捗管理なし

属人化            ←  情報が個人・チームに閉じている ←  共有の仕組みがない

改善が定着しない  ←  やらなくても誰も困らない      ←  ガバナンスの不在（最重要）
                  ←  過去の改善失敗               ←  学習性無力感
```

### 1-4. 優先課題

| 優先度 | 課題 | 理由 |
|-------|------|------|
| **1位** | 進捗不明 | チームの行動変容なしに既存データ（GitLab）から着手できる。可視化がすべての改善の前提になる |
| **2位** | 属人化 | バス係数1のリスクが高い。担当者離脱でプロジェクトが止まる |
| **3位** | 手戻り | プロセス合意が必要で、チームの当事者意識なしには解決できない。ガバナンス確立後に着手する |

## 2. ガバナンス設計

### 2-1. 問題の構造

現在のプロジェクトはガバナンスが機能していないため、「やらなくても困らない構造」になっている。推進者がチームやPMに直接働きかけても消耗するだけで組織は変わらない。

```
部長（スポンサー候補）
  ↓  ← ここへの働きかけが最優先
PM（機能不全・問題意識なし）
  ↓
PL × 4（非協力的・丸投げ志向）
  ↓
開発者・外部委託先

推進者 → チームへの直接働きかけは機能しない
推進者 → 部長への問題可視化が唯一の突破口
```

### 2-2. 部長への働きかけ計画

**目的：** 部長をスポンサーとして機能させ、PMへの指示を通じて構造的強制力を作る

**ステップ1：現状の損失を可視化する資料を作る**

感情論ではなく、以下の観点で「事実と損失」をデータで示す。

| 観点 | 示すべき内容 |
|------|------------|
| コスト損失 | 手戻りによる工数増・デスマーチ化のリスク |
| 品質リスク | テスト計画なし・デグレード検知不能の実態 |
| 組織リスク | バス係数1による担当者離脱時の影響 |
| コンプライアンスリスク | A-SPICE・ISO 26262審査時のワークプロダクト不在 |

**ステップ2：部長に求めることを明確にする**

曖昧な「改善してほしい」ではなく、具体的なアクションを依頼する。

- PMへの改善活動推進の指示
- 推進者の役割範囲の公式な承認（推進者はPM代替ではなく基盤整備者）
- 月次での改善状況の報告受け取り（部長が関与することでPMへの圧力になる）

**ステップ3：推進者の役割範囲を合意する**

| 推進者がやること | 推進者がやらないこと |
|----------------|------------------|
| 改善の仕組み・基盤の整備 | PMの代わりに進捗管理をする |
| 現状の問題可視化・報告 | チームへの直接の業務指示 |
| プロセス・テンプレートの提供 | 丸投げされた管理業務の引き受け |
| 改善状況のモニタリング | 外部委託先の直接マネジメント |

### 2-3. PMへの関与設計

部長からPMへの指示が出た後、PMがまた何もしないリスクがある。以下の仕組みで構造的に動かす。

- **月次報告の仕組みを作る** — 推進者が改善状況レポートを作成し、部長・PMに自動配信する。PMが何もしていない事実が部長に見える状態にする
- **改善会議の設定** — PM主催の改善定例会議を月次で設定し、推進者はファシリテーターとして参加（主体はPM）
- **判断を委ねない** — 推進者が「どうしましょうか」とPMに聞くのではなく、「この方針で進めます、問題あれば連絡ください」というスタイルにする

## 4. 実行計画

### 4-1. Phase別アクションプラン

| Phase | 期間 | 主なアクション | 推進者の動き | 期待成果 |
|-------|------|--------------|------------|---------|
| **Phase 0** | 1〜2週 | 部長への現状報告・スポンサー合意 | 損失可視化資料の作成・部長への報告 | 改善活動の公式承認・役割範囲の合意 |
| **Phase 1** | 2〜4週 | GitLabデータ分析・現状可視化 | 自動レポートの作成・ベースライン記録 | 進捗・担当者の概況が見える状態 |
| **Phase 2** | 1〜2ヶ月 | 工程定義ワークショップ・最小ルール合意 | ファシリテーション・テンプレート提供 | 工程定義と最小タスク管理ルールの合意 |
| **Phase 3** | 2〜3ヶ月 | 自動レポート配信・定着モニタリング | モニタリング・部長への月次報告 | PMなしでもレポートが自動配信される状態 |
| **Phase 4** | 3ヶ月〜 | 手戻り対策・要件管理プロセス整備 | プロセス設計支援 | チームが自律的に改善サイクルを回せる |

> **Phase 0を最優先にする理由：** ガバナンスが確立されていない状態でPhase 1以降に進んでも、推進者がPM化するだけで組織は変わらない。部長のスポンサー合意を得てから技術的な改善に着手する。

### 4-2. 推進者の役割範囲

```
やること（推進者の責任）
  ├── 部長への現状報告・改善状況の可視化
  ├── 自動レポート・ダッシュボードの整備
  ├── プロセス・テンプレートの設計と提供
  └── 月次改善レビューのファシリテーション

やらないこと（PMとチームの責任）
  ├── 日々の進捗管理・タスクアサイン
  ├── 外部委託先への指示・確認
  ├── チームメンバーへの直接の業務指示
  └── 丸投げされた管理業務の引き受け
```

## 5. リスクと対処

### 5-1. 主なリスク

| リスク | 発生条件 | 対処法 |
|-------|---------|-------|
| **推進者PM化** | チームの丸投げを受け入れ続ける | 役割範囲を部長と合意した上で明文化。丸投げは「それはPMの役割です」と返す |
| **部長が動かない** | 損失可視化資料が刺さらない | コスト・コンプライアンスリスクの観点を強調。数値で損失を示す |
| **Phase 0で止まる** | 部長との合意が取れない | 合意できる範囲（自動レポートのみ等）から始め、成果を見せてから次を提案する |
| **ワークショップが形骸化** | PLが参加するが何も変わらない | 部長が「ワークショップの結論をPMに報告せよ」と指示する構造を作っておく |
| **改善が推進者の離脱で止まる** | 推進者がいなくなると誰も動かない | スクリプト・テンプレート・手順書をGitLabに置き、チームで保有する |

### 5-2. 撤退基準

以下の条件が揃った場合、現在のアプローチでの改善推進は継続しない。

- [ ] 部長がスポンサーとしての役割を担わない（改善を支持する言葉はあるが指示を出さない）
- [ ] Phase 0から3ヶ月経過してもPMの行動変容が一切見られない
- [ ] 推進者への丸投げが継続し、役割の境界が守られない

> **撤退基準の意義：** 撤退基準を持つことは「諦める」ためではなく、「消耗を防ぎ、改善できる条件を上位に交渉する」ための根拠を持つためである。撤退基準に達した場合は、部長に「この条件が整わない限り改善は進まない」と伝える材料になる。
