# 期間別改善計画ガイド

活動できる期間に応じて、何を優先し・何を諦めるかの判断基準をまとめたガイド。
期間が短いほど「ガバナンス確立」と「自動化」に集中する。

---

## 1. 期間別の現実的な到達点

### 前提：改善の積み上げ構造

改善活動には依存関係がある。下の層が確立しないまま上の層に着手しても定着しない。

```
Layer 4：改善サイクルの自律化（チームが自分で回せる）
    ↑ 必要期間：6ヶ月以上の定着
Layer 3：手戻り対策（要件・レビュープロセス）
    ↑ 必要期間：プロセス合意の定着に3ヶ月以上
Layer 2：属人化・進捗管理の定着（工程定義・ドキュメント化）
    ↑ 必要期間：習慣化に2〜3ヶ月
Layer 1：可視化・自動化（既存データ活用・自動レポート）
    ↑ 必要期間：設計・導入に2〜4週間
Layer 0：ガバナンス確立（スポンサー合意・構造的強制力）
    ↑ 必要期間：1〜2週間（ただし最優先）
```

> **原則：** 期間が短いほど下の層だけに集中する。Layer 0とLayer 1だけでも、「何も見えなかった状態」から「問題が可視化された状態」に変えることができる。それだけでも十分な価値がある。

---

### 期間別の到達点マップ

| 期間 | 到達できるLayer | 到達点の概要 |
|------|--------------|------------|
| **〜3ヶ月** | Layer 0〜1 | ガバナンス確立＋現状可視化。問題が「見える」状態になる |
| **〜6ヶ月（半期）** | Layer 0〜2 | 進捗管理・属人化対策の基盤が整う。「記録が回り始める」状態になる |
| **〜1年** | Layer 0〜3 | 手戻り対策まで着手。「プロセスが機能し始める」状態になる |
| **1年以上** | Layer 0〜4 | 改善サイクルが自律化。「チームが自分で改善できる」状態になる |

---

## 2. 期間ごとの優先施策

### 〜3ヶ月（短期集中）

**目標：** 「何も見えない状態」を「問題が可視化された状態」に変える

**選ぶ理由：** 期間が短い場合、チームの行動変容を求める施策は定着しない。既存データを使った自動化だけが確実に成果を出せる。

## 3. 期間が短い場合の判断基準

期間が制約されている場合、施策の選定は以下の基準で判断する。

### 優先する施策の条件（すべて満たすものを選ぶ）

```
① チームの行動変容が少なくても成果が出る
  └ 既存データの自動集計・可視化など

② 推進者がいなくなっても動き続ける
  └ 自動化・テンプレート・チェックリストなど仕組み化できるもの

③ 次の改善活動の土台になる
  └ ベースライン記録・属人化マップ・工程定義など
```

### 諦める施策の条件（以下に当てはまるものは後回し）

```
① 定着に3ヶ月以上かかる
  └ 文化・習慣の変化を伴うもの

② 前提となる施策が未完了
  └ 工程定義が未完成なのに手戻り対策など

③ 推進者の継続的な関与が必要
  └ 推進者がいなくなると止まる施策
```

### 期間別の優先マトリクス

| 施策 | 〜3ヶ月 | 〜6ヶ月 | 〜1年 | 1年以上 |
|-----|--------|--------|------|--------|
| ガバナンス確立（G0） | ◎必須 | ◎必須 | ◎必須 | ◎必須 |
| 自動レポート | ◎必須 | ◎必須 | ◎必須 | ◎必須 |
| 担当者マップ | ◎必須 | ◎必須 | ◎必須 | ◎必須 |
| タスク管理ルール合意 | △時間次第 | ○実施 | ○実施 | ○実施 |
| 外部委託先指示テンプレート | ◎必須 | ◎必須 | ◎必須 | ◎必須 |
| 工程定義ワークショップ | ✕見送り | ○実施 | ○実施 | ○実施 |
| 属人化対策（ペアレビュー等） | ✕見送り | △初期のみ | ○実施 | ○実施 |
| 要件テンプレート | ✕見送り | ✕見送り | ○実施 | ○実施 |
| インターフェース定義 | ✕見送り | ✕見送り | ○実施 | ○実施 |
| レビュープロセス定義 | ✕見送り | ✕見送り | ○実施 | ○実施 |
| 改善サイクル自律化 | ✕見送り | ✕見送り | △準備のみ | ○実施 |

## 4. 活動終了時の引き継ぎ設計

### 引き継ぎの原則

> **期間限定の活動で最も避けるべき失敗は「推進者がいなくなった瞬間に何も残らない状態」。**
> 仕組みをGitLabに置き、チームで保有することが最重要。

### 引き継ぎで必ず残すもの

| 成果物 | 保存場所 | 継続担当 |
|-------|---------|---------|
| 自動レポートのスクリプト・設定 | GitLabリポジトリ | PM（推進者が不在でも動く自動化） |
| 改善計画ガイド・課題別解決策ガイド | GitLabリポジトリ | PM・PL |
| 工程定義ドキュメント | GitLab Wiki | PL（各チームが更新） |
| 担当者マップ（最新版） | GitLabリポジトリ | 自動更新スクリプト |
| ベースラインと現状値の記録 | GitLabリポジトリ | 推進者の後任 |
| 月次改善レビューの議事録 | GitLab Wiki | PM（主催）・PL（参加） |

### 引き継ぎドキュメントの構成

活動終了の2〜4週間前に以下を作成する。

```
## 1. 活動のサマリー
   - 実施した施策と結果（ベースラインとの比較）
   - 定着したこと・定着しなかったこと

## 2. 継続すべきこと（次の担当者へ）
   - 自動化されていて引き続き動く仕組みの一覧
   - 人が継続的にやる必要があること（頻度・担当）

## 3. 未完了の課題と優先度
   - 着手できなかった課題の一覧
   - 次の改善活動で優先すべき順序

## 4. 注意事項・学び
   - やってみて分かった失敗パターン
   - 次の推進者が避けるべきこと

## 5. 連絡先・引き継ぎ先
   - 各チームPLの連絡先
   - 自動化スクリプトの管理者
   - 次の改善推進者（未定の場合はPMが暫定）
```

### 推進者の離脱を想定した設計チェックリスト

活動開始時から以下を意識して設計する。

- [ ] 自動化スクリプトはGitLabに置き、コメントで動作を説明している
- [ ] テンプレート・チェックリストはチームが単独で使える形式になっている
- [ ] 月次改善レビューはPM主催で推進者なしでも開催できる設計になっている
- [ ] 週次レポートは推進者がいなくても自動配信される
- [ ] 工程定義・タスク管理ルールはドキュメント化されGitLabにある
- [ ] 後任者が「どこから始めればいいか」が引き継ぎドキュメントを見れば分かる

## 付録：期間の見積もり方

改善活動の期間が未定の場合、以下の基準で必要期間を逆算する。

| 到達したい状態 | 必要な最低期間 |
|-------------|-------------|
| 現状が可視化された（自動レポートが動いている） | 2〜3ヶ月 |
| タスク管理が回り始めた（Issue更新率50%以上） | 4〜5ヶ月 |
| 工程定義が合意され使われている | 5〜6ヶ月 |
| 手戻りが減り始めた（件数が計測できている） | 8〜10ヶ月 |
| 推進者なしでプロセスが回る | 12〜18ヶ月 |

> **注意：** これは「最低期間」であり、ガバナンスが機能しない・チームの抵抗が強い場合はさらに時間がかかる。期間が短い場合は到達点の期待値を下げることが重要。
