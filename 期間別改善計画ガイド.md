# 期間別改善計画ガイド

活動できる期間に応じて、何を優先し・何を諦めるかの判断基準をまとめたガイド。
期間が短いほど「ガバナンス確立」と「自動化」に集中する。

---

## 1. 期間別の現実的な到達点

### 前提：改善の積み上げ構造

改善活動には依存関係がある。下の層が確立しないまま上の層に着手しても定着しない。

```
Layer 4：改善サイクルの自律化（チームが自分で回せる）
    ↑ 必要期間：6ヶ月以上の定着
Layer 3：手戻り対策（要件・レビュープロセス）
    ↑ 必要期間：プロセス合意の定着に3ヶ月以上
Layer 2：属人化・進捗管理の定着（工程定義・ドキュメント化）
    ↑ 必要期間：習慣化に2〜3ヶ月
Layer 1：可視化・自動化（既存データ活用・自動レポート）
    ↑ 必要期間：設計・導入に2〜4週間
Layer 0：ガバナンス確立（スポンサー合意・構造的強制力）
    ↑ 必要期間：1〜2週間（ただし最優先）
```

> **原則：** 期間が短いほど下の層だけに集中する。Layer 0とLayer 1だけでも、「何も見えなかった状態」から「問題が可視化された状態」に変えることができる。それだけでも十分な価値がある。

---

### 期間別の到達点マップ

| 期間 | 到達できるLayer | 到達点の概要 |
|------|--------------|------------|
| **〜3ヶ月** | Layer 0〜1 | ガバナンス確立＋現状可視化。問題が「見える」状態になる |
| **〜6ヶ月（半期）** | Layer 0〜2 | 進捗管理・属人化対策の基盤が整う。「記録が回り始める」状態になる |
| **〜1年** | Layer 0〜3 | 手戻り対策まで着手。「プロセスが機能し始める」状態になる |
| **1年以上** | Layer 0〜4 | 改善サイクルが自律化。「チームが自分で改善できる」状態になる |

---

## 2. 期間ごとの優先施策

### 〜3ヶ月（短期集中）

**目標：** 「何も見えない状態」を「問題が可視化された状態」に変える

**選ぶ理由：** 期間が短い場合、チームの行動変容を求める施策は定着しない。既存データを使った自動化だけが確実に成果を出せる。

```
Week 1-2：Phase 0（ガバナンス確立）
  ├── 現状損失の可視化資料作成
  └── 部長へのスポンサー合意・推進者役割範囲の確定

Week 3-4：現状調査・ベースライン記録
  ├── GitLabコミット履歴の分析
  ├── 属人化リスクマップの作成
  └── 課題のベースライン値を記録（手戻り件数・Issue更新率等）

Week 5-8：自動レポート・担当者マップの構築
  ├── 週次進捗レポートの自動生成・Teams配信
  ├── 担当者マップの自動生成
  └── 部長・PMへの配信開始

Week 9-12：定着確認・引き継ぎ準備
  ├── 自動レポートが正常に機能しているか確認
  ├── 問題が可視化されたことをPM・部長に報告
  └── 引き継ぎドキュメントの作成
```

**諦めること：**
- 工程定義ワークショップ（時間が足りない・定着しない）
- タスク管理ルールの変更（行動変容は3ヶ月では定着しない）
- 手戻り対策（Layer 2が整っていないため時期尚早）

**この期間で残せるもの：**
- 週次自動レポートの仕組み（推進者がいなくても動く）
- 現状のベースライン記録（次の改善活動の出発点）
- 属人化リスクマップ（次の担当者が優先課題を把握できる）

---

### 〜6ヶ月（半期）

**目標：** 「記録が回り始める」状態を作る

**選ぶ理由：** 半期あれば習慣化に必要な2〜3ヶ月を確保できる。自動化で土台を作った後、最小限のプロセス合意に着手できる。

```
Month 1：Phase 0（ガバナンス確立）+ 現状調査
  ├── 部長へのスポンサー合意
  ├── GitLabデータ分析・ベースライン記録
  └── 週次自動レポートの構築・配信開始

Month 2：自動化の安定化＋最小タスク管理ルールの合意
  ├── 自動レポートの精度確認・修正
  ├── タスク管理ルールのワークショップ（全チームPL）
  └── GitLab Issue最小利用ルールの展開

Month 3：タスク管理定着の支援
  ├── Issue更新率のモニタリング
  ├── 躓いているチームへの個別サポート
  └── PLへの週次リマインダー・フォロー

Month 4：工程定義ワークショップの実施
  ├── 工程定義ワークショップ（全チームPL参加）
  ├── 完了条件の合意・ドキュメント化
  └── 合意内容をIssueテンプレートに反映

Month 5：属人化対策の初期着手
  ├── 担当者マップを元にリスク高箇所の特定
  ├── ペアレビューのルール化
  └── GitLab Wikiへの最小ドキュメント化開始

Month 6：定着確認・引き継ぎ設計
  ├── メトリクスでベースラインと比較・効果を確認
  ├── 定着していない施策の原因分析・修正
  └── 引き継ぎドキュメントの作成・次のフェーズ計画の策定
```

**諦めること：**
- 手戻り対策（要件テンプレート・インターフェース定義）
- 改善サイクルの自律化（Layer 3・4は時間が足りない）
- 外部委託先のプロセス統制（優先度を下げる）

**この期間で残せるもの：**
- 週次自動レポートの仕組み
- 最小タスク管理ルール・GitLab利用ルール
- 工程定義（各チームの完了条件の合意）
- 属人化リスクマップ・ペアレビューの習慣の種

---

### 〜1年

**目標：** 「プロセスが機能し始める」状態を作る

**選ぶ理由：** 1年あれば習慣化の確認と修正を経て、手戻り対策まで着手できる。ただし改善サイクルの自律化は1年では難しく、仕組みを残すことが最重要になる。

```
Q1（1〜3ヶ月）：ガバナンス確立＋可視化・自動化
  └── 〜3ヶ月の計画と同様

Q2（4〜6ヶ月）：プロセス定義・属人化対策
  └── 〜6ヶ月の Month 4〜6 と同様

Q3（7〜9ヶ月）：手戻り対策の着手
  ├── 要件記述テンプレートの作成・パイロット（1チーム）
  ├── チーム間インターフェース定義ワークショップ
  └── レビュープロセスの定義・チェックリスト整備

Q4（10〜12ヶ月）：定着化＋自律化の準備
  ├── 全チームへの要件テンプレート展開
  ├── 手戻り件数のメトリクスで効果確認
  ├── 月次改善レビューをPM主導で自律的に回す練習
  └── 推進者なしでも動く仕組みの完成・引き継ぎ
```

**諦めること：**
- 改善サイクルの完全な自律化（仕組みは作るが、自走には至らない）
- 外部委託先のプロセス統制の深化（最低限の指示テンプレートに留める）

**この期間で残せるもの：**
- 全自動レポート基盤
- 工程定義・タスク管理ルール
- 要件テンプレート・レビューチェックリスト
- 属人化リスクマップ・ドキュメント化の習慣
- 月次改善レビューの仕組み（PM主導で回せる状態）

---

### 1年以上

**目標：** 「チームが自分で改善できる」状態を作る

**選ぶ理由：** 1年以上あれば、改善サイクルを回しながらチームのスキルを引き上げ、推進者がいなくても動く組織を設計できる。

```
Year 1：〜1年の計画を実施（Layer 0〜3の確立）

Year 2以降：
  Month 13-18：改善サイクルの自律化
    ├── 振り返り（レトロスペクティブ）の習慣化
    ├── チームが自分で問題を発見・提案できる場の設計
    └── 推進者の関与を段階的に減らす

  Month 18以降：推進者の撤退・チームの自走
    ├── 推進者なしでもプロセスが維持・改善される状態の確認
    ├── 新メンバーのオンボーディングへの組み込み
    └── ナレッジのリポジトリへの集約・継続管理
```

## 3. 期間が短い場合の判断基準

期間が制約されている場合、施策の選定は以下の基準で判断する。

### 優先する施策の条件（すべて満たすものを選ぶ）

```
① チームの行動変容が少なくても成果が出る
  └ 既存データの自動集計・可視化など

② 推進者がいなくなっても動き続ける
  └ 自動化・テンプレート・チェックリストなど仕組み化できるもの

③ 次の改善活動の土台になる
  └ ベースライン記録・属人化マップ・工程定義など
```

### 諦める施策の条件（以下に当てはまるものは後回し）

```
① 定着に3ヶ月以上かかる
  └ 文化・習慣の変化を伴うもの

② 前提となる施策が未完了
  └ 工程定義が未完成なのに手戻り対策など

③ 推進者の継続的な関与が必要
  └ 推進者がいなくなると止まる施策
```

### 期間別の優先マトリクス

| 施策 | 〜3ヶ月 | 〜6ヶ月 | 〜1年 | 1年以上 |
|-----|--------|--------|------|--------|
| ガバナンス確立（G0） | ◎必須 | ◎必須 | ◎必須 | ◎必須 |
| 自動レポート | ◎必須 | ◎必須 | ◎必須 | ◎必須 |
| 担当者マップ | ◎必須 | ◎必須 | ◎必須 | ◎必須 |
| タスク管理ルール合意 | △時間次第 | ○実施 | ○実施 | ○実施 |
| 工程定義ワークショップ | ✕見送り | ○実施 | ○実施 | ○実施 |
| 属人化対策（ペアレビュー等） | ✕見送り | △初期のみ | ○実施 | ○実施 |
| 要件テンプレート | ✕見送り | ✕見送り | ○実施 | ○実施 |
| インターフェース定義 | ✕見送り | ✕見送り | ○実施 | ○実施 |
| レビュープロセス定義 | ✕見送り | ✕見送り | ○実施 | ○実施 |
| 改善サイクル自律化 | ✕見送り | ✕見送り | △準備のみ | ○実施 |

## 4. 活動終了時の引き継ぎ設計

### 引き継ぎの原則

> **期間限定の活動で最も避けるべき失敗は「推進者がいなくなった瞬間に何も残らない状態」。**
> 仕組みをGitLabに置き、チームで保有することが最重要。

### 引き継ぎで必ず残すもの

| 成果物 | 保存場所 | 継続担当 |
|-------|---------|---------|
| 自動レポートのスクリプト・設定 | GitLabリポジトリ | PM（推進者が不在でも動く自動化） |
| 改善計画ガイド・課題別解決策ガイド | GitLabリポジトリ | PM・PL |
| 工程定義ドキュメント | GitLab Wiki | PL（各チームが更新） |
| 担当者マップ（最新版） | GitLabリポジトリ | 自動更新スクリプト |
| ベースラインと現状値の記録 | GitLabリポジトリ | 推進者の後任 |
| 月次改善レビューの議事録 | GitLab Wiki | PM（主催）・PL（参加） |

### 引き継ぎドキュメントの構成

活動終了の2〜4週間前に以下を作成する。

```
## 1. 活動のサマリー
   - 実施した施策と結果（ベースラインとの比較）
   - 定着したこと・定着しなかったこと

## 2. 継続すべきこと（次の担当者へ）
   - 自動化されていて引き続き動く仕組みの一覧
   - 人が継続的にやる必要があること（頻度・担当）

## 3. 未完了の課題と優先度
   - 着手できなかった課題の一覧
   - 次の改善活動で優先すべき順序

## 4. 注意事項・学び
   - やってみて分かった失敗パターン
   - 次の推進者が避けるべきこと

## 5. 連絡先・引き継ぎ先
   - 各チームPLの連絡先
   - 自動化スクリプトの管理者
   - 次の改善推進者（未定の場合はPMが暫定）
```

### 推進者の離脱を想定した設計チェックリスト

活動開始時から以下を意識して設計する。

- [ ] 自動化スクリプトはGitLabに置き、コメントで動作を説明している
- [ ] テンプレート・チェックリストはチームが単独で使える形式になっている
- [ ] 月次改善レビューはPM主催で推進者なしでも開催できる設計になっている
- [ ] 週次レポートは推進者がいなくても自動配信される
- [ ] 工程定義・タスク管理ルールはドキュメント化されGitLabにある
- [ ] 後任者が「どこから始めればいいか」が引き継ぎドキュメントを見れば分かる

## 付録：期間の見積もり方

改善活動の期間が未定の場合、以下の基準で必要期間を逆算する。

| 到達したい状態 | 必要な最低期間 |
|-------------|-------------|
| 現状が可視化された（自動レポートが動いている） | 2〜3ヶ月 |
| タスク管理が回り始めた（Issue更新率50%以上） | 4〜5ヶ月 |
| 工程定義が合意され使われている | 5〜6ヶ月 |
| 手戻りが減り始めた（件数が計測できている） | 8〜10ヶ月 |
| 推進者なしでプロセスが回る | 12〜18ヶ月 |

> **注意：** これは「最低期間」であり、ガバナンスが機能しない・チームの抵抗が強い場合はさらに時間がかかる。期間が短い場合は到達点の期待値を下げることが重要。
