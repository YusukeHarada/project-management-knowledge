# チーム実践ガイド

改善計画に基づき、各ロールが「何を・いつ・どうやるか」を定めた実践手順書。
スキルレベルに関わらず「考えなくても動ける」ことを設計基準とする。

---

## 1. 対象者とスキルレベルの定義

本ガイドは以下の3ロールを対象とする。

| ロール | 対象者 | 想定スキルレベル |
|-------|-------|---------------|
| **推進者** | プロセス改善担当者 | プロセス設計・ファシリテーション経験あり |
| **PL** | 各チームのプロジェクトリーダー | 自チームの作業管理はできるが、全体視点・プロセス管理経験が薄い |
| **開発者** | 各チームの開発メンバー・外部委託先 | 技術スキルはあるが、タスク管理・進捗報告の習慣がない |

### スキルレベルと求めるアクション

| レベル | 特徴 | 求めること |
|-------|------|----------|
| **Lv1** | プロセスの経験が少ない | テンプレート・チェックリストに沿って動く |
| **Lv2** | 自分の作業は管理できる | タスクを自分でIssue化・更新できる |
| **Lv3** | 全体視点を持てる | 問題を発見し改善提案できる |

> **設計原則：** 開発者はLv1前提で設計する。PLはLv2を目標にする。Lv3は改善サイクルが回り始めてから育てる。

## 2. RACIマトリクス

改善活動における主要な役割分担を明確にする。ケースによって変動するが、基本構造として参照する。

### RACIの定義

| 記号 | 名称 | 意味 |
|-----|------|------|
| **R** | Responsible | 実作業者（実際に手を動かす人） |
| **A** | Accountable | 最終責任者・承認者（原則1人のみ） |
| **C** | Consulted | 意見・相談を求める相手（双方向） |
| **I** | Informed | 結果を通知する相手（一方向） |

### 活動別RACIマトリクス

| 活動 | 部長 | PM | 推進者 | PL | 開発者 |
|------|------|-----|--------|-----|--------|
| G0：部長への現状報告・改善承認取得 | **A** | C | **R** | C | - |
| 週次自動レポートの整備・配信 | I | **A** | **R** | I | - |
| 工程定義ワークショップの実施 | - | C | **R/A** | **R** | C |
| タスク（Issue）の登録・更新 | - | **A** | C | **R** | **R** |
| 担当者マップの作成・更新 | I | **A** | **R** | I | - |
| 月次改善レビューの運営・意思決定 | C | **A** | **R** | C | - |
| メトリクス計測・効果報告 | I | **A** | **R** | I | - |
| 引き継ぎドキュメントの作成 | I | **A** | **R** | I | - |

> **工程定義ワークショップのみ R/A が推進者に集中する理由：** PLは工程定義の改善対象者であり、Aを持たせると責任が形骸化する。推進者が設計・承認まで責任を持ち、PLは内容のオーナーとして実作業（R）に参加する。

### 設計上の注意点

- **PMがAを多く持つ構造はG0（ガバナンス確立）が前提。** 部長の承認によりPMが改善責任を負う状態が作られて初めて機能する。G0未達の場合、PMのAは形骸化する。
- **PM不作為時のエスカレーション：** PMがAの活動で意思決定・承認が停滞した場合、推進者は部長（C）に状況を報告しエスカレーションする。このパスを事前に部長と合意しておく（Phase 0で取り決める）。
- **PLへの負荷集中を避ける：** PLは日常の開発業務と並行して改善活動に参加する。AやRを過剰に持たせると改善活動自体が破綻するため、I・Cを基本にR/Aは限定的に割り当てる。

## 3. ロール別のやること・やらないこと

### 推進者

| やること | やらないこと |
|---------|-----------|
| 改善の仕組み・テンプレートの設計 | PMの代わりに進捗管理をする |
| 部長への現状報告・改善状況の可視化 | チームへの直接の業務指示 |
| 工程定義ワークショップのファシリテーション | 丸投げされた管理業務の引き受け |
| 自動レポートの整備・配信 | 外部委託先の直接マネジメント |
| 月次改善レビューの設計・運営 | 個別タスクのフォロー |

### PL

| やること | やらないこと |
|---------|-----------|
| チームのタスクをGitLab Issueに登録・更新する | 推進者に進捗管理を任せる |
| 週次でチームの進捗を把握・報告する | 口頭だけで完結させる |
| 工程定義ワークショップに参加・合意する | 合意したルールを無視する |
| 外部委託先への指示をテンプレートで文書化する | 口頭・メールのみで指示する |

### 開発者

| やること | やらないこと |
|---------|-----------|
| タスク開始時にIssueを確認・担当者を設定する | 口頭確認だけで作業を始める |
| 作業完了時にIssueのステータスを更新する | 「完了した気がする」で放置する |
| 仕様が不明な場合はIssueにコメントで記録する | 不明なまま進めて後から手戻りになる |
| コミットメッセージにIssue番号を含める | 何をしたか分からないコミットをする |

## 4. 最小ワークフロー

### 日次（開発者・PL）

```
作業開始時（5分）
  ├── 今日やるタスクのIssueを確認する
  ├── 担当者が自分になっているか確認する
  └── ステータスを「進行中」に更新する

作業終了時（5分）
  ├── 今日やったことをコミット（Issue番号付き）
  ├── Issueの進捗コメントを1行書く
  └── 完了したらステータスを「完了」に更新する
```

### 週次（PL）

```
月曜（15分）
  ├── チームのIssue一覧を確認する
  ├── 止まっているIssueがあれば理由を確認する
  └── 今週のタスク・優先度をチームに共有する

金曜（15分）
  ├── 今週の進捗をIssueの状態で確認する
  ├── 遅延・ブロックがあれば推進者・PMに報告する
  └── 来週分のタスクをIssue化しておく
```

### 月次（PL・推進者）

```
月次改善レビュー（60分・PM主催・推進者ファシリテート）
  ├── 各チームの進捗状況の共有（PL）
  ├── 遅延・ブロックの原因と対策の議論（PL・推進者）
  ├── プロセスの問題点の確認（全員）
  └── 翌月の改善アクションの決定（PM・PL）
```

## 5. GitLab 最小利用ルール

ツールの使い方を統一し、入力コストを最小化する。

### Issueの書き方（テンプレート）

```
タイトル：[チーム名] 作業内容を一言で

## やること
- 

## 完了条件
- 

## 関連Issue・MR
- 

## 期限
YYYY/MM/DD
```

### ステータスの定義

| ステータス | 意味 | 誰が変える |
|-----------|------|---------|
| `Open` | 未着手 | PL（Issue作成時） |
| `進行中` | 作業中 | 開発者（着手時） |
| `レビュー待ち` | 実装完了・レビュー依頼中 | 開発者 |
| `Closed` | 完了・マージ済み | PL（確認後） |

### コミットメッセージのルール

```
[Issue番号] 変更内容を一言で

例：
#123 ブートローダーの初期化処理を修正
#456 OTA受信モジュールのエラーハンドリング追加
```

## 6. 工程定義テンプレート（ワークショップ用）

Phase 2のワークショップで各チームが記入する。PLが主体となって回答し、推進者がファシリテートする。

```
チーム名：
工程名：[要件定義 / 基本設計 / 詳細設計 / 実装 / 単体テスト / 結合テスト]
記入者（PL）：
記入日：

1. 何をもって始まるか（開始条件）
   └ 前工程からどんな成果物・情報が揃ったら始めてよいか
   →

2. 何が必要か（インプット）
   └ 作業に必要なドキュメント・情報・環境
   →

3. 何を決めないといけないか（判断ポイント）
   └ 作業中に合意・確認が必要なこと
   →

4. 何をもって完了とするか（完了条件）
   └ 何が揃ったら「この工程は終わった」と言えるか
   →

5. その条件は誰がどう決めるか（承認者）
   └ 完了を承認する人・確認の方法
   →
```

## 7. よくある躓きと対処法

| 躓きパターン | 原因 | 対処法 |
|-----------|------|-------|
| Issueを作るのが面倒で口頭で済ませてしまう | 入力コストが高い・習慣がない | テンプレートを使い1分以内で作れる形にする。最初の1ヶ月はPLが代わりに作ってもよい |
| Issueのステータスを更新し忘れる | 更新を思い出すきっかけがない | 日次の終わりに「Issueを更新したか？」をチームに自動通知する（Teamsのリマインダー） |
| 完了条件が曖昧でIssueがいつまでも閉じない | 工程定義が合意されていない | 工程定義ワークショップでチームの完了条件を合意する（Phase 2） |
| 外部委託先がルールに従わない | 指示が曖昧・ルールが共有されていない | 委託先への指示テンプレートを作り、発注時に明示的にルールを共有する |
| PLがワークショップに参加するが何も変わらない | 上位からの強制力がない | 部長・PMにワークショップの結論を報告させる仕組みを作る（Phase 0で合意しておく） |

## 8. スキルアップのロードマップ

プロセス改善と並行して、チームのスキルを段階的に引き上げる。

```
Month 1-2:【基礎定着】 Lv1
  └ チェックリスト・テンプレートに沿って動ける
  └ Issueの登録・ステータス更新が習慣になる

Month 3-4:【自律化】 Lv2
  └ 自分でタスクを分解してIssue化できる
  └ 進捗を自分から報告する習慣が生まれる
  └ 週次レビューをPLが自律的に回せる

Month 5-6:【改善参加】 Lv2→Lv3
  └ プロセスの問題に自分で気づき提案できる
  └ 他メンバーをサポートできる
  └ 月次改善レビューでPLが意見を出せる

Month 6以降:【自走】 Lv3
  └ 推進者なしでプロセスが維持・改善される
  └ 新メンバーへのオンボーディングを自分たちで行える
```

## 付録：週次チェックリスト

### 開発者（毎日）

- [ ] 今日やるIssueを確認した
- [ ] Issueのステータスを「進行中」にした
- [ ] コミットメッセージにIssue番号を含めた
- [ ] 作業終了時にIssueにコメントを1行書いた
- [ ] 完了したIssueを「レビュー待ち」に変更した

### PL（週次）

- [ ] チームの全Issueの状態を確認した
- [ ] 1週間以上更新されていないIssueを確認した
- [ ] 遅延・ブロックがあれば推進者・PMに報告した
- [ ] 来週分のタスクをIssue化した
- [ ] 外部委託先の成果物・進捗を確認した

### 推進者（月次）

- [ ] 週次自動レポートが正常に配信されているか確認した
- [ ] 部長への月次報告資料を作成した
- [ ] 月次改善レビューをファシリテートした
- [ ] 改善仮説の検証状況をメトリクスで確認した
- [ ] 次月の改善アクションをドキュメントに記録した
