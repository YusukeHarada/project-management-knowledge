# 標準との対応マップ

本ガイドで実施する改善活動と、業界標準（A-SPICE・IPA）との対応を示す参照ドキュメント。
チームメンバー向けではなく、推進者・上位報告・審査対応を目的として使用する。

---

## 1. A-SPICEとは何か

### 概要

**Automotive SPICE（A-SPICE）** は、車載ソフトウェア開発のプロセス能力を評価するための国際標準モデル（ISO/IEC 33000シリーズをベースに車載向けに特化）。欧州の自動車メーカー（OEM）が自社のサプライヤーに対して適用を求めるケースが多く、日本の車載開発においても対応が求められる場面が増えている。

### 能力レベルの定義

| レベル | 名称 | 意味 | 現場での目安 |
|-------|------|------|------------|
| **Level 0** | 不完全 | プロセスが実施されていないか、目的を達成できていない | 暗黙知プロジェクトの典型状態 |
| **Level 1** | 実施済み | プロセスが実施され、成果物が存在する | 最低限の工程定義・ワークプロダクトがある状態 |
| **Level 2** | 管理済み | プロセスが計画・監視・調整されている | 進捗管理・レビュープロセスが機能している状態 |
| **Level 3** | 確立済み | 組織として標準プロセスが定義・展開されている | 組織共通のプロセスが全チームに適用されている状態 |
| **Level 4** | 予測可能 | プロセスが定量的に管理されている | メトリクスによる定量管理が機能している状態 |
| **Level 5** | 最適化（Optimizing） | プロセスが継続的に改善されている | 改善サイクルが自律的に回っている状態 |

> **現実的な目標：** 暗黙知プロジェクトからの改善では、まずLevel 0→Level 1の達成を目指す。Level 2は改善活動が1年以上定着した後に射程に入る。

### 主要なプロセスエリア

本ガイドに関連する主なプロセスエリアは以下の通り。

| プロセスエリア | 略称 | 内容 |
|-------------|------|------|
| プロジェクト管理 | MAN.3 | 計画・進捗管理・課題管理 |
| リスク管理 | MAN.5 | リスクの識別・評価・対処 |
| ソフトウェア要件分析 | SWE.1 | 要件の定義・管理 |
| ソフトウェアアーキテクチャ設計 | SWE.2 | アーキテクチャの設計・文書化 |
| ソフトウェア詳細設計・実装 | SWE.3 | 詳細設計・実装・単体テスト |
| 構成管理 | SUP.8 | 変更管理・バージョン管理 |
| 変更要求管理 | SUP.10 | 変更要求の追跡・管理 |
| プロセス改善 | PIM.3 | 改善活動の計画・実施・評価 |

## 2. IPAガイドラインとは何か

### 概要

**IPA（情報処理推進機構）** は、日本の経済産業省所管の機関であり、ソフトウェア開発の品質・生産性向上に関する調査・ガイドライン策定を行っている。主な参考資料として以下がある。

| 資料名 | 内容 | 本ガイドとの関連 |
|-------|------|---------------|
| **ソフトウェア開発データ白書** | 日本のソフトウェア開発の実態データ・ベンチマーク | 手戻りコスト・工数の根拠データとして活用可能 |
| **非機能要求グレード** | 非機能要件の定義・評価基準 | 要件テンプレートの非機能要件欄に活用 |
| **ソフトウェア開発分析データ集** | 工数・品質・生産性の統計データ | 改善効果の根拠として部長説明に活用 |
| **SLCP（共通フレーム）** | ISO/IEC 12207ベースのプロセス定義フレームワーク | 工程定義の参考モデル |

### SLCPの工程定義との対応

IPA/SECが定める共通フレーム（SLCP）の工程と、本ガイドの工程定義の対応は以下の通り。

| SLCP工程 | 本ガイドでの対応 |
|---------|--------------|
| 企画プロセス | 要件定義の前段階（スコープ・目的の合意） |
| 要件定義プロセス | 課題1「進捗不明」の要件テンプレートで対応 |
| 開発プロセス（設計・実装・テスト） | 工程定義の5つの問いで各工程を定義 |
| 保守プロセス | 属人化対策・ドキュメント化で対応 |
| 管理プロセス | 課題1「進捗不明」の解決策全般 |

### 活動とプロセスエリアの対応

| 本ガイドの活動 | 対応するA-SPICEプロセスエリア | 生成されるワークプロダクト |
|-------------|--------------------------|----------------------|
| **G0：ガバナンス確立** | MAN.3（プロジェクト管理） | 改善活動計画書・役割定義 |

| **タスク管理（GitLab Issue）** | MAN.3・SUP.10 | タスク一覧・変更要求記録 |
| **担当者マップ・属人化対策** | MAN.3・SUP.8 | 担当者マップ・構成管理計画 |
| **要件テンプレート** | SWE.1 | ソフトウェア要件仕様書 |
| **インターフェース定義** | SWE.2 | ソフトウェアアーキテクチャ仕様書 |
| **レビュープロセス定義** | SWE.3・SUP.2 | レビュー記録・検証報告書 |
| **月次改善レビュー** | PIM.3 | 改善活動報告書 |
| **メトリクス計測** | MAN.3（定量管理） | 測定記録・品質メトリクス |

### A-SPICEレベルとの対応

| レベル | 本ガイドでの到達条件 | 必要な活動 |
|-------|------------------|---------|
| **Level 0 → Level 1** | 各工程の成果物が存在する | G0＋工程定義＋要件テンプレートの適用 |
| **Level 1 → Level 2** | プロセスが計画・監視されている | タスク管理定着＋週次レポート＋月次改善レビュー |
| **Level 2 → Level 3** | 組織として標準プロセスが定義・展開されている | 全チームへのプロセス展開＋組織共通定義の確立 |

## 4. 本ガイドとIPAの対応マップ

### IPA推奨プラクティスとの対応

| IPAの推奨プラクティス | 本ガイドでの対応 |
|--------------------|--------------|
| 工程ごとの入口・出口基準の設定 | 工程定義の5つの問い |
| レビューの実施と記録 | レビュープロセス定義・チェックリスト |
| 変更管理の徹底 | GitLab Issue・変更通知ルール |
| メトリクスによる品質管理 | メトリクス定義・ベースライン計測 |
| 振り返りと改善サイクル | 月次改善レビュー・Phase 4改善サイクル |
| リスクの早期発見 | 週次レポート・担当者マップ |

### IPAデータを部長説明に活用する

IPA「ソフトウェア開発データ白書」「ソフトウェア開発分析データ集」が示す傾向を、部長への現状損失説明に活用できる。

> **注意：** 以下は業界で定性的に広く認識されている傾向であり、自社プロジェクトの実測値とは異なる場合がある。部長説明では「自社の具体的な事例・工数記録」と組み合わせて使うことで説得力が増す。数値の正確な一次出典を求められた場合は、自社データを根拠にする方が確実である。

| データ | 内容（定性的傾向） | 活用場面 |
|-------|-----------------|---------|
| **手戻りコスト** | 後工程での修正コストは前工程より大幅に増大する（発見が遅いほど修正範囲が広がるため） | 手戻り多発の損失構造を説明する |
| **要件定義の重要性** | 欠陥の多くは要件定義フェーズに起因する（上流の不明確さが下流で顕在化する） | 要件テンプレート導入の必要性を説明する |
| **レビュー効果** | 上流でのレビュー実施は後工程の手戻りを大幅に削減する（早期発見ほど修正コストが低い） | レビュープロセス整備の費用対効果を説明する |
| **属人化リスク** | 担当者離脱はプロジェクトの進行に深刻な影響を与える（知識の引き継ぎに相当の工数が必要） | バス係数1のリスクを説明する |

## 5. 部長・審査員への説明の仕方

### 部長への説明（Phase 0の現状損失資料に組み込む）

標準への言及は「権威づけ」として使う。感情論ではなく「業界標準に沿った改善」として提示することで、承認を得やすくなる。

**説明の流れ：**

```
1. 現状はA-SPICEのLevel 0（プロセスが機能していない）状態にある
   └ 具体的な根拠：進捗不明・手戻り・属人化の実態データ

2. 車載ソフト開発の業界標準ではLevel 1〜2が最低要件とされている
   └ OEMからの要求・将来の審査リスクとして提示

3. 今回の改善活動でLevel 1を目指す
   └ 本ガイドの活動がA-SPICEの要求事項に対応していることを示す

4. 改善しないまま放置した場合のリスク
   └ A-SPICE審査が入った場合、ワークプロダクトが存在せず一から作成が必要
   └ 作成コスト＋審査遅延リスクを提示
```

### 審査員への説明（将来のA-SPICE審査対応）

本ガイドの活動で生成されたワークプロダクトを、A-SPICE審査の証拠として提示できる。

| A-SPICEの確認観点 | 提示できるワークプロダクト |
|----------------|----------------------|
| プロセスが定義されているか | 工程定義ドキュメント（工程別Entry/Exit Criteria） |
| 進捗が管理されているか | 週次レポート・GitLab Issueの履歴 |
| 要件が定義・追跡されているか | 要件テンプレートで作成したIssue |
| レビューが実施されているか | レビューチェックリスト・MR記録 |
| 変更が管理されているか | GitLab Issue・コミット履歴 |
| 改善活動が行われているか | 月次改善レビュー議事録・メトリクス記録 |

> **注意：** A-SPICE審査の正式対応には本ガイドだけでは不十分な部分がある。審査が具体的に決まった段階で、A-SPICE専門家によるギャップ分析を別途実施することを推奨する。
