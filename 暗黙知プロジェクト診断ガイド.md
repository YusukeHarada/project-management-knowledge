# 暗黙知プロジェクト診断ガイド

タスク管理・スケジュール管理が機能しない組込み開発プロジェクトの実態と対処法

---

## 1. 「暗黙知プロジェクト」とは何か

タスク・スケジュール・担当者が誰かの頭の中にだけ存在し、組織の共有情報として管理されていない状態のプロジェクトを「暗黙知プロジェクト」と定義する。

> **定義：** プロジェクト情報（タスク・担当者・進捗・期限）が個人の頭の中または個人管理のExcelにのみ存在し、組織として可視化・共有されていないプロジェクト状態

プロジェクト管理ツールの不在や形骸化が主な原因であり、組込み・車載ソフト開発において特に顕著に現れる。

---

## 2. 典型的な特徴・前提・意識

### 2-1. プロジェクト構造上の特徴

- 進捗状況を正確に把握している人物が存在しない（または一人に集中している）
- WBS・タスクリストが最初から存在しないか、作成後に更新されていない
- マイルストーンが「大まかな日程」としてしか定義されておらず、中間成果物が設定されていない
- 担当者アサインメントが口頭のみで行われており、文書化されていない
- Issue管理ツール（Redmine、GitLab等）が存在するが実質的に使用されていない
- コードは書かれているがコミットメッセージに文脈情報がない

### 2-2. チーム・組織の意識・前提

- 「自分の作業は自分で分かっている」という個人最適の意識が支配的
- プロジェクト管理は「誰かがやること」という責任の曖昧さ
- ツールへの入力を「余計な作業」と認識する文化
- 「報告しなくても分かるだろう」という暗黙の期待
- PMが名目上は存在するが権限・技術力・時間のいずれかが不足している
- 外部ベンダー・オフショアへの発注内容が口頭ベース

---

## 3. よく起きる問題

### 3-1. 日常的に発生する問題

| 問題カテゴリ | 具体的な発生事象 |
|------------|----------------|
| 進捗不明 | 誰も「全体の何%完了か」を答えられない。Redmineのチケットが数ヶ月放置されている |
| 二重作業 | 同じ機能を複数人が実装していることに後から気づく |
| 手戻り多発 | 仕様認識が人によって異なり、完成品のレビューで大幅修正が発生する |
| スケジュール崩壊 | 遅延に誰も気づかないまま納期直前に発覚。バッファが全て消費されている |
| 属人化・バス係数1 | 担当者が休暇・退職するとプロジェクトが止まる |
| 品質劣化 | テスト計画が存在しないためデグレードに気づかない |
| ベンダー統制不能 | 外注先への指示が曖昧なため成果物品質が安定しない |

### 3-2. 中長期で顕在化する問題

- A-SPICE・ISO 26262審査時にワークプロダクトが存在せず一から作成が必要になる
- 新メンバーのオンボーディングコストが異常に高い（ドキュメントがないため口頭伝承）
- 技術的負債の蓄積が見えないため、リファクタリング計画が立てられない
- プロジェクト後のナレッジが残らないため、次プロジェクトで同じ失敗を繰り返す

---

## 4. 「見かけ上のメリット」と実態

暗黙知プロジェクトが継続される背景には、関係者にとって一見メリットに見える側面がある。しかし多くは短期的・個人的なものであり、組織・プロジェクト全体では損失となる。

| 見かけ上のメリット | 実態・裏に潜むリスク |
|------------------|-------------------|
| 管理の手間がかからない（ツール入力不要） | 問題が見えていないだけ。後でまとめて爆発する |
| 自由度が高く、自分のペースで作業できる | 他者との依存関係が見えず、突然ブロックされる |
| 会議・報告が少なく開発に集中できる | 認識齟齬が蓄積し、後工程で大幅手戻りが発生する |
| ベテランが裁量を持って動ける | ベテラン離脱でプロジェクトが止まる（バス係数1） |

---

## 5. 組織・プロジェクトへの影響（デメリット）

### 5-1. コスト・スケジュールへの影響

- 手戻りコスト：認識齟齬による後工程修正は前工程修正の10〜100倍のコストがかかる
- デスマーチ化：遅延が可視化されないまま進み、納期直前に全員が長時間残業を強いられる
- 見積精度ゼロ：過去データが存在しないため次回以降の見積りも根拠なしの感覚値になる

### 5-2. 品質・セキュリティへの影響

- テスト網羅性が担当者の経験と記憶に依存し、再現性がない
- 変更影響範囲の分析ができず、修正のたびにデグレードリスクが生じる
- セキュリティ脆弱性・安全要件の見落としが後から判明する

### 5-3. 組織・人材への影響

- 特定メンバーへの業務集中とバーンアウトリスク
- 若手エンジニアが「どう仕事を進めるか」を学べず成長が遅れる
- 優秀な人材が「組織として機能していない」と判断し離職する

---

## 6. AI・自動化導入時に起きうる問題

### 6-1. データ不在問題

> **注意：** AI・自動化ツールは「存在するデータ」しか処理できない。タスクが頭の中にある状態では、AIに処理させるデータそのものがない。最初の入力コストは人間が負担しなければならない。

- コミット履歴から推測できる情報は限定的（何を実装したかは分かるが、残タスクは分からない）
- 初期データ投入なしにIssue管理を始めても「空のシステム」ができるだけ

### 6-2. 導入抵抗

- 「入力が面倒」という抵抗：ツール入力が「余計な仕事」と認識される
- 「監視されている」という警戒：タスクの可視化を個人評価と結びつけて拒否反応
- 「今まで動いていた」という現状維持バイアス：機能不全の現状が「普通」として認識されている

### 6-3. ツール導入の失敗パターン

- ツールを導入したが誰も使わず形骸化（Redmineの二の舞）
- AI自動化の管理者が「実質的なPM」になってしまい、導入推進者に業務が集中する
- ツールへの入力ルールが定まらず、データの一貫性が保てない

---

## 7. 現実的な改善戦略

### 7-1. 基本原則

> **原則：** 「人間の行動変容」を最小化し、「既にある情報」を最大限活用する設計にする

- まず「ある情報」（GitLabコミット履歴・Excelスケジュール）から現状マップを自動生成する
- 人間への入力要求は「確認」レベルに留める（Yes/No、数字の修正 程度）
- 成功体験を小さく積む：週次レポートの自動化など、すぐ効果が見える施策から始める
- 導入推進者自身がPM化しないよう、「情報提供者」ではなく「基盤整備者」に役割を限定する

### 7-2. フェーズ別アプローチ

| フェーズ | 期間 | やること | 期待効果 |
|--------|------|---------|---------|
| Phase 1 | 1〜2週 | GitLabコミット解析 → 担当者マップ自動生成 | 誰が何を触っているかが見える |
| Phase 2 | 1ヶ月 | Excelスケジュールを取り込みGitLab Issueに自動変換 | タスクの最小管理基盤ができる |
| Phase 3 | 2ヶ月 | Teamsへの週次レポート自動投稿 | 進捗が自動で共有される |
| Phase 4 | 3ヶ月〜 | 遅延検知・A-SPICEワークプロダクト自動生成 | PMレス運用が成立する |

### 7-3. 導入推進者が陥りやすい罠

> **注意：** 自動化基盤を整備した人が「分かる人」として頼られ、実質的にPMの役割を引き受けるリスクがある。「ツールを作る人」と「ツールを使う人」の境界を明確にし、権限と責任を委譲する設計が重要。

- Claude Codeによる自動化スクリプトはリポジトリに置き、チームで保有する（個人資産にしない）
- 週次レポートの受信者は推進者本人ではなく、プロジェクトオーナー・上長に設定する
- 「可視化レポートを渡す」→「判断・改善はチームが行う」という構造を明確にする

- [ ] 現時点でプロジェクト全体の進捗率を即答できる人物がいない
- [ ] タスクリスト・WBSが存在しないか、最後に更新されたのが3ヶ月以上前
- [ ] Issue管理ツールのチケットが1ヶ月以上更新されていないものがある
- [ ] 担当者アサインが口頭のみで文書化されていない
- [ ] スケジュール遅延を検知する仕組みが存在しない
- [ ] プロジェクト全体を把握しているのが1〜2人のみ（バス係数1）
- [ ] 議事録が存在しないか、作成されてもアクションアイテムが追跡されていない
- [ ] テスト計画書が存在しないかコードと乖離している
- [ ] 外部ベンダーへの指示が口頭・メールのみで管理されていない
- [ ] 過去プロジェクトの振り返り情報が存在せず、同じ問題が繰り返されている
